# Патч для усиления глав 4-5: архитектурная доказательность и воспроизводимость

## Вставка в главу 4: корректность включения внешнего результата

Внешний результат считается корректно включённым в экосистему FuzzyXAI, если адаптер либо строит валидный объяснительный объект `E_k`, удовлетворяющий обязательным полям ExplainPlan, либо возвращает диагностическое состояние `D_k`. Дальнейшие операторы применяются только к валидным объектам или к явно зафиксированным диагностическим ветвям.

Корректное включение требует выполнения следующих условий:

| Условие | Смысл |
| --- | --- |
| Сохранение источника | Каждый канал, влияющий на действие, имеет трассируемый источник. |
| Сохранение семантики | Выход внешнего метода не подменяется новой интерпретацией без адаптера. |
| Построение термов | Числовые значения переводятся в `L_k` и `mu_k` по ExplainPlan. |
| Правиловая проверяемость | Правила `R_k` и активации `alpha_k` фиксируются явно. |
| Учёт неопределённости | Неопределённость либо представлена, либо признана непокрытой. |
| Действие после наблюдателя | Автоматическое действие вычисляется только после риск-наблюдателя. |

## Вставка в главу 4: инварианты маршрута

1. **Инвариант сохранения источника.** Каждый объект, влияющий на действие, имеет трассируемый источник `tau_k`.
2. **Инвариант запрета скрытой редукции.** Представление неопределённости не упрощается без вычисления потери `Delta`.
3. **Инвариант запрета прямого действия.** Ни один внешний результат не допускается к автоматическому действию без риск-наблюдателя.
4. **Инвариант диагностической замкнутости.** Если объект не может быть построен или согласован, система возвращает диагностическое состояние, а не продолжает маршрут неявно.

## Вставка в главу 5: общий протокол апробации

1. Фиксируется входной артефакт и trace.
2. Фиксируется ExplainPlan, включая веса, пороги и правила действия.
3. Запускаются адаптеры внешних моделей или объяснителей.
4. Строятся `E_k` или диагностические состояния `D_k`.
5. Выполняются операторы согласования, выбора представления, редукции и риск-наблюдения.
6. Формируется действие: `accept`, `lower_confidence`, `request_more_data`, `defer_to_human` или `block`.
7. Результат сравнивается с базовым режимом.
8. Сохраняется доказательный пакет с hash ExplainPlan, значениями операторов, диагностикой и итоговым действием.

## Угрозы валидности

| Тип угрозы | Ограничение |
| --- | --- |
| Внутренняя валидность | Корректность вычисления операторов зависит от ExplainPlan и функций принадлежности. |
| Внешняя валидность | Контрольные сценарии не означают промышленной или клинической сертификации. |
| Конструктная валидность | Диагностический разрыв не равен предметной ошибке модели. |
| Воспроизводимость | Требуются фиксированные версии данных, ExplainPlan и кода. |
| Субъективность порогов | `gamma_max`, `Delta_max` и веса риска должны калиброваться или задаваться регламентом. |
