# Дорожная карта развития FuzzyXAI

Этот файл фиксирует самые перспективные направления проекта и переводит их в план реальной разработки. Цель: довести прототип глав 2-3 до состояния, которое можно показывать как исследовательскую платформу, библиотеку и инженерный инструмент для аудита объяснимости.

## Главная идея продукта

FuzzyXAI не должен конкурировать с моделью классификации и не должен заменять SHAP/LIME. Его сильная позиция другая:

- модель отвечает на вопрос: **какой риск и какие признаки важны?**
- локальное объяснение отвечает: **почему модель дала этот риск?**
- системный оператор отвечает: **не сломалось ли объяснение при переходе от модели к решению, протоколу, врачу, аудитору или отчёту?**

То есть продуктовая формула:

```text
ML model + local explanation + fuzzy system operator = проверяемая цепочка объяснимости
```

## Самые перспективные направления

### 1. Оператор как middleware для AI-конвейеров

Суть: сделать оператор не только demo-функцией, а промежуточным слоем между компонентами системы.

Пример цепочки:

```text
модель риска -> модуль правил -> клинический протокол -> текстовый отчёт -> аудит
```

Что должно работать:

- каждый компонент отдаёт `ExplanationObject`;
- `compose(...)` проверяет совместимость соседних объяснений;
- если всё согласовано, считается `I(E_G)`;
- если язык объяснений разошёлся, возвращается `D_ij`;
- GUI показывает, на каком ребре и почему цепочка сломалась.

Почему это необычно: большинство XAI-инструментов объясняет одну модель, а здесь проверяется объяснимость всей системы.

### 2. Benchmark “без оператора / с оператором”

Суть: показывать ценность оператора не абстрактно, а через сравнение.

Baseline без оператора:

- есть `predict_proba`;
- есть accuracy / ROC-AUC;
- есть feature importance;
- нет проверки согласованности между компонентами;
- нет `gamma`, `I(E_G)`, `D_ij`.

С оператором:

- сохраняется риск модели;
- строится `E_k`;
- выбирается представление `A_k^F`;
- проверяется композиция `model -> decision`;
- появляется диагностика семантического разрыва.

Минимальный готовый benchmark уже есть в `apps/defense_demo.py` на `sklearn breast_cancer` и `RandomForestClassifier`.

Следующий шаг: вынести это в отдельный воспроизводимый benchmark-скрипт с несколькими моделями и искусственными типами разрыва.

### 3. Библиотека в стиле scikit-learn

Суть: инженер должен использовать проект в 5-10 строк.

Целевой API:

```python
from fuzzyxai import FuzzyXAIPipeline

pipe = FuzzyXAIPipeline.from_data(X_train, y_train, mode="audit")
result = pipe.explain_model_case(model, X_test.iloc[0])
result.plot_chain()
print(result.report)
```

Что нужно довести:

- единый API для модели, риска и объяснения;
- адаптеры для `sklearn`, `xgboost`, `lightgbm`;
- сериализация `ExplainPlan` в JSON/YAML;
- стабильные типы результата;
- документация с 3-5 короткими примерами.

### 4. ExplainPlan Builder для эксперта

Суть: врач или аналитик должен видеть, как числа превращаются в термы.

Нужно:

- загрузка CSV;
- автоопределение числовых и категориальных признаков;
- построение `low / medium / high` по квантилям;
- ручная настройка границ термов;
- граф функций принадлежности;
- экспорт `ExplainPlan`;
- объяснение простыми словами, что именно меняется.

Почему важно: без этого оператор выглядит математической абстракцией. С этим появляется инженерия знаний.

### 5. Адаптивная калибровка весов

Суть: веса `beta`, `lambda`, `eta`, `kappa` не должны выглядеть выбранными вручную.

План:

- собрать пары объяснений `E_i, E_j`;
- задать экспертную оценку рассогласования `r_expert`;
- обучить веса через оптимизацию;
- сравнить ручные и обученные веса;
- показать граф сходимости и ошибку на валидации.

Практичный вариант:

- сначала `scipy.optimize` или projected gradient;
- затем `optuna` для красивого поиска;
- сохранить отчёт в `reports/calibration_*`.

### 6. Диагностика типов семантического разрыва

Суть: `D_ij` должен быть не просто “ошибка”, а понятный диагноз.

Типы разрыва:

- нет общих термов;
- совпали термы, но разные активные правила;
- близкий риск, но разные решения;
- высокая неопределённость одного компонента;
- потеря редукции слишком велика;
- конфликт источников в `NeutrosophicFS`;
- неполный trace `tau`.

В GUI нужно показывать:

- ребро, где произошёл разрыв;
- тип разрыва;
- вклад в `gamma`;
- рекомендацию: “проверьте термы”, “проверьте правило”, “нужен аудит источника”.

### 7. Многоуровневое представление как “линза аудитора”

Суть: глава 3 должна выглядеть не как классификация типов множеств, а как выбор линзы под ситуацию.

Пример:

- для пациента достаточно `F0` или `IntervalFS`;
- для врача нужен `HesitantFS`, если есть мнения экспертов;
- для аудитора нужен `MultiLevelFS`, если есть конфликт источников и trace;
- для регулятора нужен полный отчёт с потерей редукции.

Нужно развить:

- profile builder `P_sit`;
- визуальный Pareto-выбор;
- рекомендации “почему выбран именно этот класс”;
- режимы `user`, `doctor`, `audit`, `regulator`.

### 8. Отчёт для защиты и аудита

Суть: после запуска GUI должен формироваться нормальный отчёт.

В отчёте:

- модель и датасет;
- выбранный кейс;
- риск модели;
- функции принадлежности;
- выбранный `A_k^F`;
- `gamma`, `I(E_G)`, `D_ij`;
- таблица вкладов в рассогласование;
- JSON trace для воспроизводимости.

Форматы:

- HTML для браузера;
- JSON для машинной проверки;
- PDF через печать страницы;
- Markdown для вставки в диссертацию.

### 9. Контролируемая LLM-генерация объяснений

Суть: LLM не должна “придумывать медицину”, она должна только переводить уже рассчитанные значения в понятный текст.

Правила:

- температура `0`;
- промпт сохраняется в trace;
- LLM получает только структурированный отчёт;
- текст должен ссылаться на `risk`, `gamma`, `uncertainty`, `D_ij`;
- если есть `D_ij`, LLM не имеет права писать “решение согласовано”.

Это перспективно, потому что соединяет строгий оператор и человеко-читаемый отчёт.

## Критерии готовности

Для защиты:

- GUI запускается одной командой;
- есть реальная модель и открытый датасет;
- виден режим “без оператора / с оператором”;
- `D_ij` воспроизводимо появляется при конфликте;
- `gamma` раскладывается на компоненты;
- README и презентационный сценарий понятны на русском;
- тесты проходят локально.

Для публикации/репозитория:

- API не зависит от GUI;
- есть стабильные примеры;
- отчёты воспроизводятся скриптами;
- есть CI;
- есть roadmap и ограничения применимости.

Для реального пилота:

- доменный датасет;
- экспертная разметка рассогласований;
- калибровка весов;
- аудит безопасности текста;
- валидация с врачами или экспертами предметной области.

## Что говорить коротко

Главная формулировка:

> Мы не просто объясняем одну модель. Мы проверяем, сохраняется ли смысл объяснения при переходе между компонентами интеллектуальной системы.

Ещё короче:

> SHAP показывает важные признаки. Наш оператор показывает, не сломалась ли объяснительная цепочка.
